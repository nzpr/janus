Defense in depth for Codex prompts

A local proxy that redacts only the secrets you specify before prompts leave the machine.

Janus Proxy is distributed as @nzpr/janus. It adds a narrow, deliberate security layer in front of the Responses API. It accepts only POST /v1/responses with no query string, sanitizes the JSON body using explicit socket-fed secret values, injects the bearer credential itself, and rejects everything else.

Why it exists

Security is the product, not a side effect.

Explicit redaction

The proxy only redacts values that another local process explicitly pushes over its Unix socket. If no secrets are supplied, nothing in the prompt body is altered.

Narrow attack surface

It only permits one upstream shape: POST /v1/responses with no query string. Everything else gets a fast 403 Forbidden.

Credential isolation

Callers do not supply the outgoing authorization header. The proxy injects it from stdin by default or from auth.json, and applies memory-hardening measures such as sensitive header handling and locked credential memory on UNIX.

Request flow

A thin barrier between user prompts and the network.

01

Receive locally

The proxy listens on a loopback port and is intended to sit on the same host as Codex, rather than acting as a shared network service.

02

Apply socket secrets

Before forwarding, the JSON request body is sanitized using only the explicit secrets currently loaded through the Unix socket.

03

Forward one route

Valid traffic is forwarded upstream with the expected bearer header. Extra routes, arbitrary methods, and any query string are not part of the contract.

Practical usage

Simple to run, intentionally strict in behavior.

Install npm 
npm install -g @nzpr/janus
Start the proxy auth.json + socket 
janus --auth-json \
--secret-socket /tmp/codex-secrets.sock \
--http-shutdown \
--server-info /tmp/server-info.json
Point Codex at it local provider 
PROXY_PORT=$(jq .port /tmp/server-info.json)
PROXY_BASE_URL="http://127.0.0.1:${PROXY_PORT}"

codex exec \
  -c "model_providers.openai-proxy={ name = 'OpenAI Proxy', base_url = '${PROXY_BASE_URL}/v1', wire_api='responses' }" \
  -c model_provider="openai-proxy" \
  'Your prompt here'

The intended model is a local privileged process with access to the credential and an optional sidecar process that feeds secret values into the socket. Codex itself just points at the local port.

What it does not try to be

Not a general API gateway. Not a feature buffet.

Purpose-built

This proxy is tightly scoped around one job: redacting explicit outbound secret values before prompts reach the Responses API.

Opinionated by design

Restricting methods, routes, and auth handling reduces ambiguity. That constraint is part of the security model.
